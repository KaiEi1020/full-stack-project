## MODIFIED Requirements

### Requirement: 系统必须提供 RESTful 简历接口
系统 MUST 继续通过标准 HTTP RESTful 接口提供简历资源及其相关候选人、上传、状态流转能力；在招聘领域重构后，这些接口 MUST 统一归属到 `recruitment/api` 层中的控制器实现，但不得因目录重组而改变既有业务语义。对于列表型查询接口，系统 MUST 使用统一分页能力，并以 `xxxPage` 语义命名分页查询入口。

#### Scenario: 迁移简历接口控制器归属
- **WHEN** 系统将简历、候选人或上传相关接口迁移到 `recruitment` 领域模块
- **THEN** 控制器实现必须放入 `recruitment/api` 层，并保持接口仍以 REST 方式暴露

#### Scenario: 创建或更新简历资源
- **WHEN** 客户端向简历或候选人相关资源发送合法的创建、更新或状态修改请求
- **THEN** 系统必须在保持既有 REST 语义的前提下完成处理，而不能因模块重构改成非 REST 交互模式

#### Scenario: 查询简历详情
- **WHEN** 客户端使用资源标识查询简历详情或候选人详情
- **THEN** 系统必须返回与重构前等价的资源结果或明确的不存在结果

#### Scenario: 查询职位分页列表
- **WHEN** 客户端请求职位列表这类分页查询接口
- **THEN** 系统必须通过 `xxxPage` 语义的接口返回统一分页结果，而不能继续返回无分页语义的原始数组

### Requirement: REST 接口必须返回明确的 HTTP 语义
系统 MUST 在招聘领域重构前后保持符合 RESTful 风格的 HTTP 方法、状态码与 JSON 响应结构；内部目录、模块归属和用例编排方式可以变化，但对外可观察的 HTTP 语义 MUST 保持稳定，除非变更被显式声明为破坏性调整。对于已声明为分页改造的列表接口，系统 MUST 统一返回 `{ total, list, hasNext }` 分页数据结构。

#### Scenario: 查询不存在的招聘资源
- **WHEN** 客户端请求不存在的简历、候选人或上传资源
- **THEN** 系统必须继续返回明确的资源不存在 HTTP 状态，而不是因重构产生框架异常或内部错误响应

#### Scenario: 提交非法招聘请求数据
- **WHEN** 客户端向招聘领域 REST 接口提交不满足约束的请求体
- **THEN** 系统必须继续返回可识别的请求无效状态与错误信息，而不是因 DTO 或模块迁移导致校验行为缺失

#### Scenario: 状态流转展示与操作一致
- **WHEN** 用户在招聘相关页面查看或修改简历、候选人或投递状态
- **THEN** 后端接口必须继续返回足以表达当前状态与操作结果的稳定字段，使前端无需因目录重构而改变基础交互模型

#### Scenario: 分页列表返回统一结构
- **WHEN** 客户端请求已接入统一分页能力的招聘列表接口
- **THEN** 系统必须返回包含 `total`、`list` 和 `hasNext` 的分页结果结构
