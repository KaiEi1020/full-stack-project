## Context

当前有效的职位列表链路并不是旧的 `RecruitmentController -> RecruitmentService`，而是 `JobController -> JobService -> JobRepository -> MikroOrmJobRepository`。现状中列表查询直接返回数组，没有统一分页入参/出参模型；如果后续每个模块各自实现分页，会继续产生命名、DTO 和查询实现分散的问题。

## Goals / Non-Goals

**Goals:**
- 为后端沉淀一套可复用的分页能力，包括统一请求模型、统一返回模型与通用仓储查询模式。
- 将现有职位列表接口调整为分页接口，并统一命名为 `xxxPage` 风格。
- 固定分页入参 `{ pageNo, pageSize }` 与返回结构 `{ total, list, hasNext }`，便于前后端复用。
- 让后续分页接口可以直接复用通用能力，而不是重复定义 DTO 和分页逻辑。

**Non-Goals:**
- 本次不处理复杂筛选、排序、游标分页等高级查询能力。
- 本次不强制一次性改造所有历史列表接口，只覆盖当前变更范围内的职位列表接口，并建立后续复用规范。
- 本次不引入新的外部分页库，优先基于现有 Nest + MikroORM 能力实现。

## Decisions

### 1. 统一以 `xxxPage` 作为分页接口语义命名
- 决定：控制器方法、应用服务方法以及后续分页查询接口统一使用 `xxxPage` 命名，例如 `jobPage`。
- 原因：`listXxx` 无法表达分页语义，后续维护时也难以区分是否支持分页。
- 备选方案：保留 `listXxx`，通过参数区分分页与非分页。未采用，因为语义不清晰，且会让调用方误判返回结构。

### 2. 抽取通用分页请求/响应模型
- 决定：在公共层提供分页请求 DTO 与分页响应类型，统一约束 `pageNo`、`pageSize`、`total`、`list`、`hasNext`。
- 原因：分页协议是跨模块复用能力，放在公共层比散落在业务模块中更稳定。
- 备选方案：每个模块自行声明分页 DTO。未采用，因为会造成重复字段、校验规则和 Swagger 定义分裂。

### 3. 在仓储层沉淀通用分页查询模式
- 决定：在招聘模块的仓储接口中新增分页查询方法，底层基于 MikroORM 的查询结果与总数统计组合返回分页结果。
- 原因：分页不仅是控制器出参变化，还需要数据库层原生支持 `offset/limit + total`，应在仓储层形成稳定契约。
- 备选方案：先查全部再在 service 层切片。未采用，因为数据量增长后性能和内存占用不可接受，也不具备通用复用价值。

### 4. 优先改造当前有效的职位列表链路
- 决定：本次以 `backend/src/modules/recruitment/api/controller/job.controller.ts` 及其对应 service/repository 实现为主，而不是围绕已注释、已失效的旧链路改造。
- 原因：变更应该落在真实生效路径上，避免只修改历史残留代码却不影响运行时行为。
- 备选方案：同时重构旧的 `RecruitmentController`。未采用，因为该链路当前不是实际入口，会扩大变更面。

## Risks / Trade-offs

- [接口破坏性变更] 现有职位列表的请求方式和返回结构会变化 → 通过 spec 明确标记 BREAKING，并同步更新测试与调用方。
- [复用设计过早抽象] 通用分页能力如果设计过重，后续小接口接入成本反而增加 → 首版仅固定最小协议，不加入筛选、排序、游标等扩展点。
- [仓储契约扩散] 一旦多个 repository 都接入分页，接口调整面会变大 → 先在职位查询链路验证模式，再按同一契约扩展到其他分页接口。
- [Swagger/校验一致性] DTO、接口文档和运行时返回若不同步，前端会误用 → 统一从公共 DTO/类型出发，并补齐接口测试。
